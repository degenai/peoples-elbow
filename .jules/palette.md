## 2025-04-09 - Native Button vs Div for Mobile Menu Toggle
**Learning:** When adding accessibility to legacy custom components (like a mobile menu toggle), it's highly preferred to refactor `div` elements into semantic `<button>` elements rather than adding `role="button"` and `tabindex="0"`. Native `<button>` elements inherently handle keyboard interactions (Space and Enter keys) and focus states, reducing custom JavaScript and styling overhead.
**Action:** Prioritize transforming interactive `div`s into `<button>`s with necessary CSS resets (`background: none; border: none; padding: 0; color: inherit; font: inherit;`) over piling ARIA attributes onto non-semantic elements.

## 2026-06-15 - Explicit Form Labels vs Template Cloning Logic
**Learning:** Adding `for` and `id` attributes to form labels is standard accessibility hygiene. However, in JavaScript-heavy applications that dynamically clone `<template>` elements (like the CRM contact rows), hardcoding `id`s inside the template creates duplicate IDs across the DOM. This causes the explicit `for` attribute to always target the *first* cloned instance, breaking core functionality (e.g., selecting a "Primary" contact in subsequent rows).
**Action:** When updating HTML for accessibility in a dynamic app, check if the targeted elements reside inside a `<template>`. If they do, either rely on implicit label wrapping (`<label><input>...</label>`) which handles DOM cloning gracefully, or update the JavaScript to inject dynamically unique IDs during the clone step. Avoid hardcoded IDs in repeating templates.
