## 2026-04-09 - [DOM-based XSS Risk in escapeHtml]
**Vulnerability:** The application used a DOM-based `escapeHtml` function across multiple files (`js/crm/renderer.js`, `js/crm/utility.js`, `js/d1-changelog.js`) that created a `div`, set `textContent`, and returned `innerHTML`.
**Learning:** This approach, while convenient, relies on the browser DOM to sanitize strings. It is generally safer and more predictable (especially outside browser contexts or dealing with non-strings) to use a pure string replacement method to encode specific unsafe characters (&, <, >, ", ). Furthermore, setting `textContent` and reading `innerHTML` might lead to mutation XSS (mXSS) depending on browser quirks or future browser updates. It also failed to handle `null` or `undefined` specifically, returning empty strings instead of safely coercing them.
**Prevention:** Implement a standard regex-based `escapeHtml` function globally or use a trusted sanitization library to replace specific unsafe characters. Ensure the function explicitly handles null/undefined correctly by checking `text === null || text === undefined`.

## 2026-06-15 - [DOM-based XSS Risk in main.js event arrays]
**Vulnerability:** The application stored event data inside a JS array in `js/main.js` and interpolated the object properties directly into an HTML string assigned to `innerHTML` without sanitization. Even though the data is hardcoded in the array right now, it creates a risky pattern if the array is ever populated from an external API or user input in the future.
**Learning:** Hardcoded data arrays often act as placeholders for future dynamic data sources. Code that interpolates this data into `innerHTML` must use a local `escapeHtml` function to safely render the properties and prevent DOM-based XSS, even if the current data appears "safe". Additionally, when applying HTML escaping, hardcoded HTML entities (like `&middot;`) in the source data should be replaced with literal characters to prevent double-encoding.
**Prevention:** Always sanitize data extracted from objects or arrays before interpolating it into HTML strings for `innerHTML` injection, and establish a local HTML encoding utility where one does not already exist.

## 2026-06-17 - Incomplete Custom HTML Sanitizer in components.js
**Vulnerability:** The manual DOM-based HTML sanitizer in `js/components.js` (`sanitizeHTML`) only removed `<script>` tags, `on*` attributes, and `javascript:` URIs.
**Learning:** Custom blocklist sanitizers are fundamentally incomplete and bypassable by numerous XSS vectors (e.g., `<object>`, `<embed>`, SVG injections) that aren't explicitly caught.
**Prevention:** Rely on established, actively maintained sanitization libraries like DOMPurify to parse and clean HTML, defaulting to an empty string (fail-closed) if the library fails to load.

## 2026-06-17 - DOM XSS via innerHTML in js/d1-changelog.js
**Vulnerability:** DOM-based XSS via unescaped `innerHTML` in `js/d1-changelog.js`.
**Learning:** Directly assigning user-controlled strings (like commit messages) to `innerHTML` allows arbitrary code execution. The presence of formatting replacement (e.g., regex replacements for bolding labels) often masks the severity and discourages using safer alternatives like `textContent`.
**Prevention:** Eliminate `innerHTML` entirely when dealing with data retrieved from external sources or databases. Construct DOM nodes using safe APIs (`document.createElement`, `textContent`, `append`) to render dynamic content, and implement formatting features programmatically (e.g., using text nodes and specific element nodes) rather than string interpolation.
