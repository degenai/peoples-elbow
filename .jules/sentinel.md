## 2024-05-25 - [DOM-based XSS Risk in escapeHtml]
**Vulnerability:** The application used a DOM-based `escapeHtml` function across multiple files (`js/crm/renderer.js`, `js/crm/utility.js`, `js/d1-changelog.js`) that created a `div`, set `textContent`, and returned `innerHTML`.
**Learning:** This approach, while convenient, relies on the browser DOM to sanitize strings. It is generally safer and more predictable (especially outside browser contexts or dealing with non-strings) to use a pure string replacement method with regular expressions to encode specific unsafe characters (&, <, >, ", ). Furthermore, setting `textContent` and reading `innerHTML` might lead to mutation XSS (mXSS) depending on browser quirks or future browser updates. It also failed to handle `null` or `undefined` specifically, returning empty strings instead of safely coercing them.
**Prevention:** Implement a standard regex-based `escapeHtml` function globally or use a trusted sanitization library to replace specific unsafe characters. Ensure the function explicitly handles null/undefined correctly by checking `text === null || text === undefined`.

## 2025-04-12 - [Overly Permissive CORS Origin Validation]
**Vulnerability:** The `admin-auth-worker.js` and `crm-worker.js` implementations used `origin.endsWith('.peoples-elbow.pages.dev')` to allow Cloudflare Pages PR previews.
**Learning:** Using `endsWith()` for origin checks is inherently risky. While `.peoples-elbow.pages.dev` with the leading dot correctly prevents trivial spoofing via external domains (e.g., `not-peoples-elbow.pages.dev`), an attacker could still exploit this within the *same* Cloudflare Pages platform by submitting a malicious branch name (e.g. `feat/auth` -> `feat-auth`) or by somehow controlling a subdomain that matches the pattern if branch names aren't strictly sanitized by the platform itself. It is safer to use a strict regex.
**Prevention:** When validating dynamic origins (like PR previews), always use a strict regular expression that perfectly describes the expected format (e.g., `/^https:\/\/[a-zA-Z0-9-]+\.peoples-elbow\.pages\.dev$/`) instead of loose string matching methods like `endsWith()` or `includes()`.
